# STM32 兼容性重构

将旧工程视为可执行规格，而不是可随意参考的代码。目标是先实现可验证的
行为等价，再改善结构；不能因为“代码更现代”而改变上电、通信或保护行为。

## 强制边界

- 未得到明确授权时，旧工程 Boot 与 App 只读；不得修改、格式化、复制回写或
  用 Git 操作覆盖它们。
- 先确认目标范围：仅迁移 App、仅重做 Boot，还是同时重做两者。不要默认扩大。
- 不把“能编译”或“单次能启动”当作兼容完成。必须覆盖上电、升级、通信、异常
  复位和关机等可观察行为。
- 任何寄存器、延时、协议或链接地址的改动，都必须说明它改变的兼容性契约。
- JLink 默认只做非侵入读取；不要无提示执行 Reset、Halt、Run、写 Flash 或写
  寄存器。调试暂停可能断开 CAN、冻结外设或改变机器电源状态。

## 先读的参考

- 处理 Boot/App 跳转、IAP 或分区时，先读
  [references/boot-app-handoff.md](references/boot-app-handoff.md)。
- 设计测试、排查“旧工程正常而新工程异常”时，先读
  [references/verification-matrix.md](references/verification-matrix.md)。
- 需要落到目录、CMake 和 RT-Thread 架构时，使用 `stm32-cmake-arch` skill。
- CAN IAP 帧格式本身是问题时，再使用项目专用的 `can-iap-protocol` skill。

## 工作流

### 1. 建立只读基线

1. 标出旧 Boot、旧 App、新 Boot、新 App、共享 IAP 和链接脚本的准确路径。
2. 记录芯片、Flash/RAM/CCMRAM 分区、向量表地址、编译器、优化级别、时钟源、
   RTOS 与升级入口。
3. 从旧工程提取行为清单：GPIO 初始电平、外设时钟、DMA/IRQ 所有权、任务周期、
   协议帧、超时、错误恢复和可见灯/蜂鸣效果。
4. 为每一项写下证据来源：源码、原机日志、逻辑分析仪、上位机抓帧或非侵入 JLink
   读数。源码推断必须标为“待上机验证”。

### 2. 写兼容性账本

为每个系统行为建立一行账本，至少包含：

| 领域 | 旧行为 | 新实现位置 | 证据 | 验证状态 |
| --- | --- | --- | --- | --- |
| Boot/App 交接 | VTOR、MSP、外设残留、升级标志 | 链接脚本/启动代码 | 源码或寄存器 | 未测/通过 |
| 时间基准 | SysTick、RT tick、DMA/定时器时钟 | system/Core | 代码+测量 | 未测/通过 |
| 外设 | GPIO、CAN、UART、ADC、PWM、I2C | driver | 寄存器/波形 | 未测/通过 |
| 协议 | ID、方向、RTR、DLC、字节序、周期 | app/task | 抓帧 | 未测/通过 |
| 业务 | 开关机、充电、急停、保护、重启 | task | 现场步骤 | 未测/通过 |

不要一次性重构多个账本项。每次只修改一个可验证的差异，并立即更新状态。

### 3. 先让最小 App 在旧 Boot 下工作

当目标是新 App 兼容旧 Boot 时，保持旧 Boot 不动，按以下顺序推进：

1. 链接地址、向量表、MSP、`SystemInit` 和 C 运行时初始化正确。
2. 保留 Boot 已配置的硬件状态；不要把 Cube 的全量 GPIO 初始化当作无害操作。
3. 先验证时钟、SysTick、RTOS 调度、堆和中断嵌套。
4. 再逐个接入 GPIO、CAN、UART、ADC、PWM、I2C 等驱动。
5. 最后接入任务状态机、周期上报、远程命令和异常恢复。

### 4. 处理 StdPeriphLib 到 HAL 的差异

- 显式检查 HAL 初始化结果；不要假设 Cube 生成配置与旧工程等价。
- 明确每个外设由谁拥有：直接寄存器、HAL IRQ、DMA 回调和 RTOS 线程不能竞争同一
  接收/状态机资源。
- 保留旧的延时语义。`rt_thread_delay()` 是 tick，`rt_thread_mdelay()` 是毫秒；
  要检查 tick 频率和所有累加计数器的单位。
- Cube 生成代码可以在 USER CODE 区或经过记录的最小补丁中适配；不要为了“零改动”
  而放弃必要的 IRQ、VTOR 或启动修复。
- 启动顺序、GPIO 上电电平、DMA 优先级、IRQ 优先级和滤波器配置都是行为，不是
  风格问题。

### 5. 先验证协议，再判断业务是否坏了

对每个上位机命令同时核对：PNG/ID、Target/Local、IDE、RTR、DLC、Data、字节序、
回应类型和回应周期。出现“能周期上报、请求无回应”时，优先检查：

1. UART/CAN RX IRQ 是否实际把字节交给旧协议解析路径；
2. HAL 与手写 IRQ/DMA 是否竞争同一标志或数据寄存器；
3. 远程帧的 RTR、DLC、地址是否与路由条件一致；
4. 回应是否被错误发到 CAN、UART 或被 DMA busy 丢弃；
5. 新 App 是否只实现了周期上报而漏掉远程请求分支。

### 6. 以现场行为收尾

依次执行验证矩阵中的场景，并记录结果。至少包括：冷启动、旧 Boot 跳新 App、
正常开机、急停、充电接入/拔出、上位机请求、IAP、看门狗复位和低功耗关机。

构建通过后再烧录；烧录通过后再宣称硬件行为通过。

## 重做 Boot 的额外规则

只有在旧 App 已在旧 Boot 下稳定兼容后才开始重做 Boot。把新 Boot 当作独立迁移：

1. 先冻结 Boot/App 交接契约和 IAP 协议，不改变 App。
2. 在硬件上对比旧 Boot 与新 Boot 的跳转前状态、跳转动作和 App 首秒行为。
3. 保留可回退的旧 Boot 镜像与恢复烧录步骤。
4. 不要清零所有外设或 GPIO，除非契约明确要求；这可能改变升级中的机器状态。
5. 将“Boot 留在升级模式”和“Boot 跳 App”的判定、标志擦除时机、无效 App 的处理
   分别测试。

## 完成条件

- 兼容性账本没有未知的关键启动、通信或安全项。
- 构建、静态检查和目标板验证均有证据。
- 原项目未被修改。
- 所有有意差异均已记录，并得到用户确认。
